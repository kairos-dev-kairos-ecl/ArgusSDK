<objective>
Close findings F6 and F7: unify the delivery contract and eliminate silent batch truncation.

1. F6 (high): splunk (connector.go:246-274) and elastic (connector.go:271-307) return DeliveryAck{Status:"failed"} with a NIL Go error for HTTP-level failures (unreadable body, unparseable response, HEC code != 0, bulk errors:true, non-2xx). Dispatcher.process (connector.go:265-285) only checks `err != nil`, so these failures are invisible — never logged, never counted. Locked contract: any non-delivered outcome returns a non-nil error AND a failed ack; the ack carries detail. Kafka already conforms — do not touch kafka in this plan.
2. F6 (counters): Dispatcher has accepted/delivered/failed/dropped fields that are never incremented. Wire accepted in Enqueue, delivered/failed in process.
3. F7 (high): splunk (connector.go:168-172) and elastic (connector.go:210-214) silently truncate `signals = signals[:limit]` and report "delivered" for the dropped remainder. Locked fix: chunk into ceil(n/limit) sequential requests; first failed chunk aborts and returns error; ack aggregates.

Constraints (from 03-CONTEXT.md, binding):
- No new dependencies. No proto / pkg/signal API changes.
- Sequential execution on the main tree — note that plan 03-03 also edits elastic/connector.go and runs AFTER this plan.
- Keep the existing behavior that signals failing OCSF mapping are skipped (mapping failure is not a delivery failure); an entirely-unmappable batch still returns Status="delivered" with empty payload.

Purpose: Delivery failures currently vanish — operators see neither logs nor counters, and oversized batches lose data while reporting success. This is the visibility half of the data-loss story (03-01 is the durability half).
Output: chunked, contract-conforming splunk + elastic Send; wired dispatcher counters with Stats(); tests for all of it.
</objective>
