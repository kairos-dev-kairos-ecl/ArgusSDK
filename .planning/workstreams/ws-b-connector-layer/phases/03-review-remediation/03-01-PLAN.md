<objective>
Rewrite the WAL/buffer layer to close the five buffer findings from the 2026-06-10 review (F1, F2, F3, F5, F14). These findings share one code path and are fixed in a single coherent pass:

1. F1 (critical): the consumed marker stomps the first byte of the 4-byte length prefix, corrupting the stream for payloads >= 16 MB. New record format: `[1-byte status][4-byte BE length][payload]`; markConsumed writes only the status byte.
2. F2 (critical): data race — Write releases segMu then reads b.seg while rotation can concurrently close/replace it.
3. F3 (critical): drain only reads the current segment; rotated segments are stranded forever (silent data loss) and never deleted (unbounded disk growth); countDropped never incremented.
4. F5 (critical): agent.stop passes a nil drain func to Flush → nil deref panic on first buffered record.
5. F14 (medium): backoff sleeps up to BackoffMax inside the streamRecords callback while the segment file handle is open.

- agent.go changes are limited to the F5 nil-drain guard in stop(); do NOT implement agent.start wiring (registration, dispatcher creation, buffer.Start call).
- Windows: tests MUST Close() all file handles before delete/rename (Windows file locking) — pattern already established in buffer_test.go.
- Sequential execution on the main tree at C:\Users\Drupad\argus-sdk — no worktree isolation.

Purpose: The dead-letter buffer is the data-loss safety net for every connector. Until F1-F3 are fixed it actively loses data; until F5 is fixed graceful shutdown panics.
